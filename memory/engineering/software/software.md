# Cross-cutting principles

These hold across all software work; the language / build / test nodes below are specific instantiations of them.

- **Mechanics to tools, judgment to me.** Let automated tools own the mechanical and enforceable — formatting, naming, lint, type-checks — and spend effort on design. **Respond to a tool's output instead of fighting it:** when it prescribes a fix, apply *that* fix; suppress a finding only when genuinely unavoidable, and only with existing precedent. (Distinct from a tool's *substantive claims* — those I always verify independently; see working-style.)
- **Judge a tooling or config change by its effect on real code, not in the abstract.** Choosing *which* mechanics to enforce is judgment (the enforcing is mechanics, above): adopt or keep a rule only when it measurably improves the actual codebase's readability, safety, or simplicity — validate against real, substantial code (the more complex, the sharper the test) and wind back what doesn't earn its place.
- **Optimize for the reader; the obvious place is the right place.** Put each thing where a reader will first look, and make that guess correct. Design and document to the interface/contract — a reader should know what something is *for* and how to use it without reading the body.
- **Name from the thing's own perspective, so the name spares the reader the body.** A name states what the entity intrinsically *is* or *does* — its role or action — and nothing incidental: not its type (`mTopic`, not `mTopicString`); not the surrounding context it sits in (a free function is a context-independent unit — leaking the caller's domain into its name or parameters means it should be a method, or the name is wrong); not a generic placeholder (`file`/`base`/`pointer` make the reader chase what they hold); not a property of the result (name the action, not the quality it yields, e.g. not "effective…"). Cut vestigial and redundant words — one verb, no `…To` tail. Parameters and locals answer to the same test.
- **Single source of truth; derive, don't store.** Keep exactly one authoritative copy of a fact or resource and compute the rest from it; don't cache what can be derived. Prefer the *effective, consumer-visible* form as that copy — derive persisted and exposed views from it (re-emit from the decoded model rather than keeping the raw input), and let data no consumer can reach fall away. Ownership boundaries are explicit, and a component never reaches outside its own (one owner sets things up; the rest defer). Where a fact genuinely can't be single-sourced — a tool version named in build config, in prose docs, and in a symlink's target — a change must sweep **every** replica; enumerate the representations a content search never surfaces (filenames, symlink targets, pinned submodule/version pointers) before calling the sweep complete.
- **Composition is realized via version-controlled submodules (the general concept is in `engineering`).** The components are repositories the root pulls in as **flat, non-recursive submodules** — one copy each, never recursing into a submodule's own submodules. So the **complete state of a project = the root's checkout + its flat submodule checkouts**, and a submodule's own tooling/infra is dormant when it is not the root. Shared infra propagates to a consumer by bumping the submodule pointer. The root gates all infra/tooling setup on a top-level check, realized per build system (see cmake).
- **A transformation's output is not guaranteed valid input — migrate onto the new form, don't round-trip the old through it.** A producer's output can be rejected by its own consumer, and pushing an old artifact through a newer version silently carries stale/renamed entries and drops values. To move state across versions (config, schema, any serialize↔deserialize), build the target from its own authoritative template and port values over deliberately — map renames, drop removals, validate against the consumer — and capture any baseline in isolation, since a context-discovering tool contaminates a "pristine" snapshot. Across a version boundary be liberal — tolerate and ignore unknown/extra fields rather than rejecting (forward/backward compat) — while still flagging an unknown a human supplied directly, since that is a typo, not an evolution.
- **Explicitness over cleverness; everything must earn its place.** Spell things out at the point of use; add an alias, layer, or abstraction only when it removes real duplication or genuinely aids comprehension — never to save keystrokes. By the same test, what has *stopped* earning its place goes: remove dead or unused code, config, and comments aggressively — once you've confirmed they are genuinely dead, not load-bearing structure that merely looks vestigial.
- **Encode the contract in the structure.** Make the type/interface state the truth — ownership, lifetime, invariants, error modes — so misuse is hard and intent is self-evident. Depend on contracts, not implementations.
- **Logic lives in named units; boundary glue stays thin.** Keep callbacks, handlers, registration, and wiring to a thin forward into a named function, and size that function to grow — assume any handler may become substantial, so give it a name and a home from the start instead of inlining logic at the glue site.
- **A teaching artifact installs a mental model — keep its own domain invisible.** When the deliverable's purpose is to teach a system (example, tutorial, reference), aim every choice at what's being taught, not the example's domain: pick a domain familiar enough to need no explanation (and get its facts right), choose names whose nouns already carry each concept's role so behaviour is inferable without reading bodies, make the structure the smallest that still exhibits the system's real value, and keep prose on the contract. The domain is scaffolding, never the subject.
